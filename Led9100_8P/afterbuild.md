# 编译后处理流程

本流程适用于 `Led9100_8P` 工程。每次 Keil 编译完成后，按以下顺序处理 HEX 和 BIN 编译产物。

## 1. 确认编译成功

- 仅当编译结果为 `0 Error(s)` 时执行后续流程。
- 编译失败时停止处理，不清理、不重命名、不移动产物。

## 2. 获取产品名前缀

- 根据当前实际生效的芯片配置，读取对应的 `HARDWAREVERSION`。
- 取硬件识别字符串开头的 7 个字符作为产品名前缀。
- 不得直接使用源码中出现的第一条 `HARDWAREVERSION`，必须结合当前宏配置判断实际生效项。

示例：

| `HARDWAREVERSION` | 产品名前缀 |
|---|---|
| `LED9100_8P_V1.01` | `LED9100` |
| `LED9101_8P_V1.01` | `LED9101` |
| `LED6800_8P_V1.01` | `LED6800` |

## 3. 获取编译日期

- 使用本次成功编译的日期。
- 日期格式为 `YYMMDD`，例如 `2026-07-11` 写作 `260711`。

## 4. 执行清理脚本

每次成功编译后都必须在 `Led9100_8P` 固件根目录下执行清理脚本，不得跳过：

```bat
keilkill.bat
```

该脚本会递归删除 Keil 中间文件、`cache` 和 `Backup` 目录，但保留 `.axf`、`.hex` 和 `.bin`。脚本包含 `pause`，执行时必须提供按键输入使其正常结束，并确认退出状态；清理失败时停止后续产物处理并报告。

## 5. 获取原始产物

当前 Keil 工程输出名为 `led9100ciu`，成功编译后需要处理以下两个文件：

```text
Led9100ciu\MDK\Objects\led9100ciu.bin
Led9100ciu\MDK\Objects\led9100ciu.hex
```

其中 `.hex` 由 Keil 的 `CreateHexFile` 生成；`.bin` 由 After Build 命令生成：

```text
fromelf --bin ".\Objects\@L.axf" --output ".\Objects\@L.bin"
```

`@L` 会按当前工程输出名展开为 `led9100ciu`。任一文件缺失或大小为 0 时停止移动并报告，不得只交付其中一个文件。

## 6. 生成目标文件名

基础格式：

```text
{产品名前缀}ciu_{编译日期}.{扩展名}
```

例如：

```text
LED9100ciu_260711.bin
LED9100ciu_260711.hex
```

## 7. 处理同名产物

- 如果固件根目录中两个基础文件名均未被占用，直接使用基础文件名。
- 如果任一基础文件名已存在，不得覆盖已有产物。
- 从 `_2` 开始查找 `.bin`、`.hex` 都未被占用的最小滚码。
- 同一次编译生成的 `.bin`、`.hex` 必须使用相同滚码。

示例：

```text
LED9100ciu_260711_2.bin
LED9100ciu_260711_2.hex
```

再次重名时依次使用 `_3`、`_4`、`_5` 等。

## 8. 移动到项目根目录

将重命名后的两个文件全部移动到 `Led9100_8P` 固件根目录：

```text
D:\03 hengrui_2025\led9100_8P\Led9100_8P
```

不要把最终交付产物只留在 `Led9100ciu\MDK\Objects` 中。

## 9. 最终校验与报告

处理完成后确认：

- 两个文件都存在于 `Led9100_8P` 固件根目录。
- 两个文件的产品名前缀、日期和滚码完全一致。
- 两个文件大小均大于 0。
- 原始 `led9100ciu.bin` 和 `led9100ciu.hex` 已完成本次移动。
- 报告最终文件名、完整路径和文件大小。
