# 开发规范

## 编码规则
- C 语言源文件和头文件（`.c`、`.h`）：统一使用 UTF-8 (no BOM) 编码。
- Python、Java 及其他编程语言项目：统一使用 UTF-8 (no BOM) 编码。
- Markdown 文档（`.md`）：统一使用 UTF-8 (no BOM) 编码。

## 执行要求
- 新建文件时必须按上述编码创建。
- 修改已有 C/H 文件时，保存结果必须为 UTF-8 (no BOM)；厂商库或历史文件若暂未修改，可保留原编码，避免无关的批量转码。
- 禁止在同一项目中混用多种文本编码。

## 提交前检查
- 提交前确认新增/修改文件编码符合规则。
- 提交前确认 C/H 文件可由当前 Keil 工具链正常编译，中文注释显示正常。
- 若发现乱码，先修复编码再提交。

## C 代码格式
- 所有以 `/* ... */` 形式注释掉的备注不得修改，包括内容、标点、换行和缩进。
- 模板工程中已有的中文备注不得修改，不做润色、不改字、不重排。
- `if` / `else` / `while` / `for` / `switch`等控制语句换行写左大括号时，左右大括号相对控制语句缩进一级。
- 大括号内部的功能语句与左右大括号对齐，不再额外缩进一级。
- 如果{}中的内容是空的，或者只有1句代码，保留左右大括号和控制语句在同一行。
- 不使用常见的 Allman “大括号后内容再深一层但括号本身与 if 对齐”的其他变体。
- 示例：

```c
for (i = 0; i < (uint32_t)len; i++)
    {
    std_usart_tx_write_data(PRINT_COM, (uint32_t)ptr[i]);
    while (!std_usart_get_flag(PRINT_COM, USART_FLAG_TC)) {}
    }
```
- 嵌套结构逐层增加缩进。
- 当一个代码块内既有普通语句，又嵌套新的 `if` / `else` / `while` / `for` 控制块时，普通语句和嵌套控制块之间空一行，用来分隔“先执行的动作”和“后续条件处理”。不是所有嵌套控制块前都机械空行，只有前面已经有实际执行语句时才空一行。
- 语句末尾使用英文分号 `;`，不得混入中文分号 `；`。
示例：

```c	
if (std_usart_get_cr1_interrupt_enable(UART3, USART_CR1_INTERRUPT_RXNE))
    {
    if (std_usart_get_flag(UART3, USART_FLAG_RXNE))
        {
        UartRxBuf3[rx_ptr3] = (uint8_t)std_usart_rx_read_data(UART3);
        
        if (rx_ptr3 < BUFF_SIZE)
            {
            rx_ptr3++;
            }
        TimOut1mS[TTCPIN3] = 0;
        }
    }
```
## GitHub版本管理

### 分支策略
- `main`：可发布分支，只放可编译、可交付代码。
- `feature/*`：功能开发分支，例如 `feature/bt-wakeup`。
- `fix/*`：问题修复分支，例如 `fix/uart-parse`。
- `release/*`（可选）：版本冻结/联调分支。

### 日常流程
1. `git switch main`
2. `git pull origin main`
3. `git switch -c feature/xxx`
4. 开发后执行 `git add -A` 与 `git commit -m "type(scope): summary"`
5. `git push -u origin feature/xxx`
6. 自测通过后合并到 `main`（PR 或本地合并）。

### 提交信息规范
- 格式：`type(scope): summary`
- 类型：`feat`、`fix`、`refactor`、`docs`、`chore`

### 发布与Tag
1. 在 `main` 确认可编译、可运行。
2. `git tag -a vX.Y.Z -m "版本说明"`
3. `git push origin main`
4. `git push origin vX.Y.Z`

### 回滚策略
- 优先使用 `git revert <commit-id>` 回退。
- 可用 `git checkout <tag>` 查看历史版本。
- 未经确认禁止使用破坏性历史重写命令。

### 嵌入式项目约束
- 不提交编译产物和 IDE 缓存文件。
- 硬件脚位变更必须同步更新代码与文档。
- 每次发布前至少完成一次完整编译和基础回归。
